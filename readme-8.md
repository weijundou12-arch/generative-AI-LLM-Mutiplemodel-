Week 8（面试满分答案版）
章节重点 1：为什么需要 MCP（工具标准化的根问题）

面试考点 1：MCP 解决的核心问题是什么？
满分简答：MCP 的目标是把“工具/数据源”以统一协议暴露给模型，让模型能稳定发现、描述、调用工具，并在不同客户端/模型之间复用。它解决的是 工具生态碎片化、参数格式不统一、集成成本高、难治理 的问题。

面试考点 2：MCP 和你自己写的 Tools Registry 有什么区别？
满分简答：Tools Registry 是“应用内部约定”；MCP 是“跨应用/跨语言/跨平台的协议化接口”。MCP 强调 标准化描述（schema）、可发现（list tools）、统一调用（call tool）、上下文传递与安全边界，更适合企业工具平台化。

面试考点 3：为什么“标准化工具描述”很重要？
满分简答：因为模型需要可靠地理解工具能力与参数结构，否则会出现“选错工具/参数乱填”。标准化描述（name/description/input schema/output）能提高工具调用成功率，也便于自动测试与版本管理。

章节重点 2：MCP 的基本交互（你面试要能讲清协议思路）

面试考点 4：一个 MCP 工具服务至少要支持哪些能力？
满分简答：至少要支持：
1）工具发现：列出工具清单与 schema
2）工具调用：按 schema 接收参数并返回结构化结果
3）错误处理：标准错误码/错误信息
4）权限与隔离：鉴权、allowlist、审计日志（企业必备）

面试考点 5：MCP 为什么能提升可治理性？
满分简答：因为工具调用被协议化后可以统一做：鉴权、限流、审计、版本控制、灰度发布、黑名单、数据脱敏。否则工具散落在各个应用里难以管控。

章节重点 3：安全边界（企业高频追问）

面试考点 6：为什么要把工具放到“独立进程/服务”而不是直接在 Agent 内部调用？
满分简答：隔离风险：工具可能访问文件/数据库/网络。独立服务能做权限最小化、资源限额、沙箱与审计；Agent 只拿到“结果”，降低被 prompt injection 诱导执行危险操作的概率。

面试考点 7：prompt injection 在 MCP/RAG 工具链里怎么防？
满分简答：核心是“不信任检索到的文本”。把工具参数生成与内容证据隔离：工具调用要经过 schema 校验、allowlist、策略过滤；对于外部内容只能作为证据，不允许直接变成指令执行。

章节重点 4：Swarm（多 Agent 协作框架）是什么

面试考点 8：Swarm 解决什么问题？
满分简答：Swarm 解决复杂任务的“分工协作”：把不同能力拆成多个 Agent（例如 Retriever、Analyst、Writer、Critic），通过 handoff/route 把任务流转。它提升模块化、可解释性与可维护性，也更方便评测与迭代。

面试考点 9：Swarm 与“一个大 Agent”相比优势是什么？
满分简答：一个大 Agent 容易提示词臃肿、行为不可控；Swarm 通过角色分离让每个 Agent 目标更单一、工具更少、输出更稳定，并能对每段链路单独监控与优化。

面试考点 10：多 Agent handoff 的触发条件怎么设计？
满分简答：基于任务类型（检索/计算/写作）、置信度（证据不足→交给 Retriever）、以及中间产物（抽取到数字→交给 Analyst 计算）。工程上常用规则路由 + LLM 路由混合。

章节重点 5：把 Week7 Agent “平台化”的关键点

面试考点 11：把 Agent + Tools 做成平台，你会怎么做版本与变更管理？
满分简答：工具 schema 版本化（v1/v2）、灰度发布、自动回归测试（用固定用例验证工具输出稳定），以及对工具调用做审计与告警（异常调用/失败率上升）。

面试考点 12：你如何评测 MCP + Swarm 系统？
满分简答：分层评测：

工具层：成功率/延迟/错误码分布

Agent 层：任务完成率、平均 handoff 次数、引用命中率

端到端：用户满意度、成本、延迟、故障恢复能力
